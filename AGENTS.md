# AGENTS.md

本文件约束本仓库中与课程内容相关的智能体行为。本课程是“现代程序设计”课程，重点训练学生使用 Shell、Git、VSCode、Python 环境和 AI Coding Agent 完成可验证的软件开发任务。

## 课程定位

本课程用 8 周、16 课时讲解现代程序设计的基础工作流：先掌握 Bash 与 Git，再学习如何把 AI 辅助编程纳入工程过程，最后构建能够稳定产出 specification 的编写机器。

课程目标：

- 能在命令行中高效完成文件、文本、进程和项目管理任务。
- 能用 Git 组织个人开发历史，并处理基础协作场景。
- 能把 AI 工具用于理解、实现、调试、重构和测试。
- 能编写清晰、可验证、可迭代的 specification，并把它连接到代码实现和验收测试。

## 课程目录

当前课程主线：

- Shell 入门：WSL 安装、Shell 命令、文件系统、文件查看与编辑、管道与重定向、环境变量、Shell 脚本。
- VSCode：VSCode 安装、基础配置、插件管理（编程插件）、SSH 远程访问。
- Python 环境管理：Python、Conda、虚拟环境、`PYTHONPATH`、依赖安装。
- Git 版本管理：提交、分支、合并、冲突、远程协作、代码审查。
- 大模型简介：大模型基本原理、API 调用、在 Python 中调用大模型。
- AI Agent：从零开始构建一个简单 AI Agent。
- AI Coding Agent：用 AI 辅助读代码、写代码、调试、重构和补测试。
- 项目实战：从需求、specification、实现到验证的完整小项目。

## 适用范围

- `books/**/*.ipynb`：课程正文、课堂讲解、命令演示、练习与小结。
- `books/**/*.md`：课程总览、章节说明、补充阅读材料。
- `codes/`：课程代码示例或课堂练习代码。
- `slides/`：课程幻灯片素材。
- `papers/`：仅在大模型或 AI Agent 章节需要参考论文时使用。

## 总体写作原则

- 默认使用中文写作；首次出现的重要英文术语可在括号中保留英文原文。
- 面向初学者，优先讲清楚“为什么要这样做”和“命令会改变什么”。
- 每个 notebook 控制在 10-20 分钟阅读和练习量。
- 一节只解决一个清晰主题，避免把多个主题塞进一个 notebook。
- 示例应小而完整，优先选择能在 WSL/Ubuntu 中直接运行的命令。
- 不要默认学生熟悉 Linux、Git、Python 环境或 AI 工具。
- 如果命令可能修改文件、安装软件或删除内容，必须明确提醒风险和检查方式。

## Notebook 结构

教学型 notebook 通常包含：

1. 标题与预计阅读/练习时间。
2. 学习目标。
3. 核心概念与使用场景。
4. 最小命令示例。
5. 输出结果应该观察什么。
6. 常见误区或注意事项。
7. 小练习。
8. 小结与下一节链接。

不是每一节都必须包含代码单元。对于 Shell 课程，Markdown 中的命令块通常更适合课堂讲解；只有需要在 notebook 内直接执行时才添加代码单元。

Git 章节采用更偏练习的组织方式：讲解文字使用 Markdown cell；学生需要输入或复制的 Shell/Git 命令使用独立 code cell，并将 cell 语言标记为 Bash/Shell Script；命令输出示例继续使用 Markdown 中的 `text` 代码块；Mermaid 图继续使用 Markdown 中的 `mermaid` 代码块。涉及 `reset`、`push`、`remote` 等可能影响仓库状态的命令时，练习应明确要求在独立练习目录或测试仓库中运行，不要在课程项目仓库中直接操作。

## Markdown 规范

- 标题要直接表达教学意图，例如“命令在哪里”“文件查看与编辑”。
- 段落保持简短，优先使用短句、表格和清单。
- 命令、路径、文件名、环境变量使用反引号，例如 `pwd`、`PATH`、`./books`。
- 讲命令时尽量说明三件事：输入是什么、输出是什么、是否会修改文件或系统状态。
- 图片使用相对路径，例如 `figures/xxx.png`。
- 章节之间使用相对链接，例如 `[04_文件系统.ipynb](04_文件系统.ipynb)`。
- 思考题可使用 MyST admonition 或折叠块，但不要堆砌过多。

## 命令与代码示例规范

- Shell 示例优先面向 Ubuntu 24.04/WSL 环境。
- 示例命令应尽量可复制运行。
- 避免使用教师机器上的绝对路径，例如 `/Users/...`。应使用 `~/course`、`./demo`、`/tmp/demo` 等通用路径。
- 删除类命令如 `rm`、覆盖重定向如 `>`、管理员命令如 `sudo`，必须说明风险。
- 安装工具时优先选择依赖简单、初学者容易验证的工具。
- 若网络、镜像源或权限问题可能影响安装，应说明“记录报错，课堂统一处理”，不要展开复杂系统配置，除非用户明确要求。

## 课程风格偏好

- 先直觉、后命令、再练习。
- 先让学生知道“这个命令解决什么问题”，再解释选项细节。
- 对比图形界面和命令行时，强调可复现、可组合、可自动化。
- 与 AI Coding Agent 相关时，强调 Shell 命令让智能体具备读写文件、运行程序、安装依赖、执行测试的能力，同时也带来操作边界和安全确认问题。
- 不使用空泛表述，如“很简单”“自行验证”。应明确告诉学生观察什么。

## 与现有项目保持一致

- 课程书稿放在 `books/` 下，按章节文件夹组织。
- Shell 入门章节使用编号文件名，新增章节时同步更新 `01_概览.ipynb` 和相邻章节的“下一节”链接。
- 本仓库使用 `book_generate.sh` 自动生成 Jupyter Book 聚合源；通常不需要手动维护 `_toc.yml`。
- 修改 notebook 后，至少运行 `jq empty <notebook>` 校验 JSON。
- 如果修改多个 notebook，批量校验 `books/**/*.ipynb`。
- 影响目录或链接时，运行 `./book_generate.sh ./books ./_build/books source` 做最小构建验证。

## 禁止事项

- 不要把课程写成命令/API 罗列。
- 不要引入与当前教学目标无关的大段工程脚手架。
- 不要假设学生已经会 Linux、Git、Python 包管理或远程服务器。
- 不要在示例中使用教师本机路径、个人用户名或无法在学生电脑上复现的环境。
- 不要无说明地引入高风险命令、长时间训练、复杂依赖或大规模下载。

## 自检清单

修改课程内容后，检查：

- 本节主题是否单一清楚？
- 是否说明了命令的输入、输出和副作用？
- 示例是否能在 WSL/Ubuntu 中复现？
- 是否避免了教师本机路径和个人环境假设？
- 相邻章节链接是否正确？
- notebook JSON 是否有效？
